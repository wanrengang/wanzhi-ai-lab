# oh-my-opencode v3.5.0-v3.5.5 版本升级深度解读

## 引言

作为 OpenCode 生态系统的核心组件,oh-my-opencode 在过去几个版本中经历了深刻的架构演进。从 v3.5.0 到最新的 v3.5.5,项目不仅在稳定性上实现了质的飞跃,更在代码质量、架构设计和用户体验等多个维度取得了显著进步。

本文将深入剖析这六个版本的迭代历程,帮助开发者理解每个版本的核心改进及其背后的技术决策。

---

## v3.5.0 — Atlas Trusts No One

### 重大架构变革

**v3.5.0** 是一个里程碑式的版本,标志着项目从功能驱动向质量驱动的战略转变。这个版本的核心主题是"信任但验证"。

#### 强制代码审查机制

此前 Atlas 代理存在一个严重问题:子代理声明"完成"即视为任务完成,缺乏验证环节。v3.5.0 引入了强制性的手动代码审查流程:

- **Read 每一个文件**:子代理修改的文件必须逐一审阅
- **交叉验证**:将声明的成果与实际情况比对
- **全面检查**:验证导入语句、逻辑流程、边界条件
- **零容忍**:禁止橡皮图章式的走过场审查

这一机制的引入,本质上是在 AI 自主性和人类监督之间找到了平衡点。

#### Boulder Continuation 修复

Boulder continuation(巨石滚动延续机制)在 v3.5.0 中得到了全面修复:

**问题诊断**:
- 会话 ID 检查缺失导致延续提示注入到错误的会话中
- `injectBoulderContinuation` 未被 await,失败时被静默吞没
- 计划路径指向不存在的 `.sisyphus/tasks/*.yaml`
- Sisyphus 会话被错误地阻止参与 boulder

**解决方案**:
```typescript
// 检查 session_ids 过滤
if (!session_ids.includes(currentSessionId)) {
  return; // 不在列表中,不予延续
}

// 等待延续注入完成
await injectBoulderContinuation(planPath);

// 修复计划路径
const planPath = '.sisyphus/plans/*.md';
```

#### 委派系统重写

整个同步委派管道经历了彻底的重构:

- **轮询机制**:从脆弱的消息计数稳定性轮询改为原生的基于完成状态的检测
- **资源清理**:子代理会话在所有退出路径上都得到正确清理
- **工具限制**:同步任务能够取回其工具权限
- **CLI 运行器**:修复了事件处理器 promise 缺少 `.catch()` 导致的挂起

#### The Great Split - 代码库大重构

这是 v3.5.0 最令人印象深刻的工程成就:**645 个文件,25+ 个巨型文件被拆解,严格执行 200 行代码限制**。

| 模块 | 重构前 | 重构后 |
|------|--------|--------|
| `src/index.ts` | 1,004 行 | 88 行 + 4 个编排文件 |
| `delegate-task/executor.ts` | 998 行 | 16 行 + 15 个模块 |
| `background-agent/manager.ts` | 1,646 行 | 核心管理器 + 20 个模块 |
| `hooks/atlas/index.ts` | 700 行 | 25 行 + 15 个模块 |
| `tools/lsp/client.ts` | 854 行 | 129 行 + 12 个模块 |

**架构价值**:
- 新增 `src/plugin/` 层:21 个文件组成插件接口
- 所有 `utils.ts` 重命名为具体的功能描述性名称
- **零破坏性变更**:所有公共 API 保持不变

这种重构体现了工程化的最高境界:在不破坏外部契约的前提下,彻底改善内部结构。

## v3.5.1-v3.5.3 — 稳定性迭代

### v3.5.1: 修复关键竞态条件

**问题**: `look_at` 工具没有等待 multimodal-looker 代理响应就返回结果。

**根因**: `df0b9f76` 提交将同步的 `session.prompt` 改为异步的 `session.promptAsync` + 轮询机制,引入了竞态条件:轮询器在服务器注册会话为忙碌状态之前就触发了。

**解决方案**: 回退到同步 prompt 以修复竞态条件。

### v3.5.2: 自动更新安全性

**关键改进**:

1. **锁定版本受到尊重**
   - 自动更新现在会跳过你显式锁定的版本
   - 安装失败时,配置 pin 会回滚以防止配置与磁盘版本不匹配

2. **子代理生命周期**
   - 修复了 `permission.question=deny` 覆盖导致的僵尸会话
   - 在 boulder 状态读取中添加可选链保护

3. **MCP 工具守护**
   - Tool after-hooks 现在安全保护 `output.output`,防止 MCP 工具返回非标准形状

### v3.5.3: 后台代理稳定性

**核心修复**:

1. **错误事件处理**
   - 后台任务现在正确处理 `session.error` 事件
   - 僵尸任务(永远占用并发槽位的任务)现在会被正确清理,包括:
     - 错误状态标记
     - 并发释放
     - 完整的定时器/队列清理

2. **认证系统增强**
   - 多层认证注入以支持桌面应用兼容性
   - 尝试 5 种策略(而非单一 `setConfig`):
     - setConfig → interceptors → fetch wrapper → mutable config → top-level fetch

3. **CLI Run 改进**
   - 会话作用域的事件订阅
   - 添加 10 秒稳定期后再检查完成,防止过早退出

## v3.5.4 — The Reliability Patch

### 可靠性大修

**v3.5.4** 是一个稳定性补丁版本,包含 30+ 修复,使 oh-my-opencode 达到日常驾驶的坚如磐石级别。

**统计数据**: 200 个文件变更,+10,921/−4,755 行代码

#### 后台代理僵死任务彻底修复

**问题**: 后台代理曾经会无声地死亡。僵死任务会像幽灵一样缠绕你的会话——已生成但从未完成,在仍在活跃运行的任务上超时。

**核心修复**:
```typescript
// 修复从杀死活跃运行的后台任务的僵死超时
// —导致代理"挂起"的核心错误

// 检测从未收到进度更新的僵死任务(真正的僵尸 vs 活跃工作者)
if (task.lastUpdate > STALE_THRESHOLD && !hasProgress) {
  markAsZombie(task);
}

// 在所有 message.part.updated 事件上刷新 lastUpdate,而不仅仅是工具事件
on('message.part.updated', () => {
  task.lastUpdate = Date.now();
});
```

#### 会话与工具继承

**新增特性**:
- 添加 `session-tools-store` 用于跟踪每个会话的工具限制
- 在后台任务通知中继承父会话工具限制
- 在所有父会话通知路径中应用 `parentTools`
- 在提示发送站点存储工具限制以确保持久执行

#### Stop Hooks 与延续修复

**关键正确性修复**:
```typescript
// 执行所有 Stop hooks,而不是在第一个非阻塞结果后返回
// (之前会在第一个非阻塞 hook 后就返回,导致后续 hooks 不执行)
for (const hook of stopHooks) {
  const result = await hook();
  if (result && result.blocking) {
    // 只有阻塞结果才中断
    break;
  }
}
```

**Todo 延续改进**:
- 为所有有未完成 todo 的会话触发 todo 延续,而不仅仅是当前会话
- 添加冷却和停滞上限以防止重新注入循环

#### 性能优化

1. **Rules injector**:
   - 基于 mtime 的解析缓存
   - 脏写守护 - 跳过冗余文件系统写入

2. **Directory injectors**:
   - 当没有注入新路径时跳过 `writeFileSync`

3. **Session messages**:
   - 基于事件的缓存,防止重复 `.messages()` 调用导致的内存泄漏

4. **Comment checker**:
   - 硬进程回收和全局信号量,防止 CPU 失控

#### Doctor 重新设计

- 3 层输出(默认/状态/详细)和整合检查
- 正确的 oMoMoMoMo 品牌,移除了过时的提供商检查

#### 社区贡献者修复 (10 位贡献者)

- **@ojh102**: 保护 against 非字符串工具输出
- **@COLDTURNIP**: 从 JSONC 配置文件加载 LSP 配置
- **@kaizen403**: 独立解析配置部分——一个无效字段不再丢弃整个配置
- **@solssak**: 检测 GPT 模型背后的代理提供商(如 litellm/ollama)
- **@willy-scr**: 修复项目技能发现的 `process.cwd()` 使用
- **@bvanderhorn**: 在 macOS 上解析符号链接 realpath
- **@raki-1203**: 正确执行所有 Stop hooks
- **@popododo0720**: 减少会话消息缓存的内存泄漏
- **@devxoul**: 在 look-at file_path 验证中阻止远程 URL
- **@G36maid**: 修复 Google Auth 文档链接

## v3.5.5 — 持续改进

### OpenCode 1.2.0+ 适配

**问题**: OpenCode 1.2.0 改变了事件流格式——后台代理因为不识别新的 `message.part.delta` 事件类型而丢失心跳。

**解决方案**:
```typescript
// 处理 message.part.delta 以进行心跳检测(OpenCode >= 1.2.0)
on('message.part.delta', () => {
  refreshHeartbeat();
});
```

**效果**: 后台任务不再在 OpenCode ≥1.2.0 上运行时显得陈旧或被过早杀死。

僵死任务守护现在还读取正确的会话状态字段,防止健康任务被错误标记为卡住的误报。

### Writing Agent: Kimi K2.5 + 反 AI Slop

**重大改进**:
- Writing 类别现在默认使用 **Kimi K2.5** 作为主要模型
- 在提示中明确烘焙了反 AI-slop 规则
- 如果你生成的文本听起来太"AI 打磨",这应该会产生明显更自然的输出

**技术价值**: 这反映了项目对输出质量的持续关注,不仅仅是"能工作",而是"工作得好"。

### Visual Engineering: Gemini 3 Pro 高变体

**模型升级**:
- Visual engineering 任务现在使用 **Gemini 3 Pro** 的 `high` 变体
- 刷新的回退链:Gemini → GLM-5 → Opus → Kimi
- 以获得更好的设计导向输出质量

## 升级建议

### 推荐升级路径

```bash
# 直接升级到最新版本
npm install -g oh-my-opencode@latest
# 或
bun install -g oh-my-opencode@latest
```

### 注意事项

1. **无需担心破坏性变更**
   - 所有版本的公共 API 都保持向后兼容
   - 内部重构不影响外部使用

2. **特别关注版本**
   - 如果正在使用 v3.4.x 或更早版本,**强烈建议**升级到 v3.5.4+
   - v3.5.4 的可靠性修复解决了很多已知问题

3. **OpenCode 1.2.0+ 用户**
   - 必须使用 v3.5.5+ 以获得完整的心跳支持
   - 否则后台任务可能会异常终止
